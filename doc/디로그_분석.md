# 디로그(D·LOG) 분석 — 장점 흡수 · 차별성 유지

> 멀티에이전트 워크플로(`dlog-analysis`)로 작성. 4개 렌즈 병렬 웹조사(features·reviews·positioning·design) → 우리 차별성 기준 종합.
> confidence(high/med/low) 표기. low 항목은 §6에서 별도 확인 필요.
> 우리 앱 3대 차별점: **AI 질문으로 감정 유도 / TourAPI 공식데이터 결합 / 혼행 후기 선순환**.

- '바쁘더라도 기록은 남기고 싶은 사람'을 겨냥한 사진 기반 자동 타임라인 일기·여행기록 앱으로, '작심삼일' 기록 포기층이 핵심 타깃이다. (high)
- 사진을 올리면 EXIF로 시간·위치·날씨가 자동 채워지고 이동 경로가 지도에 자동으로 그려지는 '입력 최소화' 설계가 핵심이다. (high)
- 감정은 '터치 한 번'으로 고르는 스낵형 기록이며, 월별 감정 통계·맞춤 감정 아이콘을 제공한다. (high)
- 회고는 타임라인·캘린더·지도 3개 뷰로 제공하고, iPhone·iPad·Mac·Android·Web 멀티기기 클라우드 동기화를 지원한다. (high)
- 기록은 기본 비공개이며, 원하는 것만 URL로 공개/공유한다. 본격적인 피드·팔로우 커뮤니티는 없다. (med)
- 평점은 iOS 약 4.8~4.9 / Android 약 4.5로 높으나 리뷰 표본은 작고(iOS 33개대), 다운로드는 Google Play '1만+' 구간(제3자 추정 약 4.4만)이다. (med)
- 수익화는 기본 무료 + 'Essential' 구독(월 약 1,800~2,500원), 구글플레이 '2025 올해의 자기계발 앱' 선정. (med)

## 3. 반드시 지킬 차별성 (Defend)

**(1) AI가 '질문'으로 감정을 끌어냄 — 디로그는 '터치/아이콘'까지만.**
디로그의 감정 기록은 기쁨·평온 같은 아이콘을 터치 한 번으로 고르는 수준이다(정량 통계는 강하나 '왜 그렇게 느꼈는지'는 비어 있음). 우리는 한 줄 메모를 단서로 AI가 동적 후속 질문을 던져 상황·감정의 맥락을 끌어낸다. 디로그가 '감정의 라벨'을 모은다면 우리는 '감정의 이야기'를 만든다 — 이게 일기 본문 품질의 격차로 직결된다.

**(2) TourAPI 공식 관광데이터 결합 — 디로그에는 없는 '장소 지식'.**
디로그는 EXIF 좌표로 위치를 '표시'할 뿐, 그 장소가 무엇인지에 대한 공식 데이터는 결합하지 않는다(리뷰에서 '한글 위치 수동검색 미지원' 불만까지 존재). 우리는 한국관광공사 TourAPI로 좌표 주변 장소를 자동 제안하고 확정하여, 일기와 후기의 '재료'로 공식 관광 데이터를 엮는다. 공모전 적합성과 국내 여행 실사용성 양쪽에서 해자가 된다.

**(3) 혼행 특화 + 장소별 공개 후기 선순환 — 디로그는 '조용한 1인 회고'에서 멈춤.**
디로그는 비공개 기본 + URL 선택 공개일 뿐 피드·커뮤니티가 없고, '혼자 여행' 단독 타깃팅도 (미검증) 명시되지 않는다. 우리는 일기는 비공개로 지키되 '장소별 후기'만 공개 게시판으로 분리해, 같은 장소를 혼자 다녀온 사람들의 후기가 묶여 보이는 선순환을 만든다. 사적 기록(일기)과 공적 가치(후기)를 한 앱에서 분리 운영하는 것이 디로그가 비워둔 자리다.

## 4. 디로그 약점에서 배우기 (Avoid)

- **한글 위치 수동검색 미지원** ('한글로는 검색이 안되네요'). → 우리 대응: EXIF 없는 사진의 fallback 수동 입력 + TourAPI `searchKeyword2`로 **한글 장소명 검색**을 1급 기능으로 지원. (Phase 3/4)
- **사진 아래 글 쓸 공간 부족 / 긴 글 제약**. → 우리 대응: 한 줄 메모는 가볍게, AI 질문·일기 본문에서 충분한 텍스트 영역 확보. '입력 최소화'와 '쓰고 싶을 때 충분히'를 양립. (Phase 6)
- **영상 업로드 미지원** 요청 반복. → 우리 대응: MVP는 사진 중심이되, `records` 스키마에 미디어 타입 컬럼을 미리 두어 향후 확장 여지 확보(당장 구현은 후순위). (Phase 2)
- **무료 300장 / Pro 5천 장 업로드 상한** 불만(헤비유저 이탈 위험). → 우리 대응: 공모전 MVP는 합리적 저장 한도 + Supabase Storage 무료 한도 점검, 상한을 명확히 고지해 기대치 관리. (Phase 7)
- **아이패드 최적화 부족**. → 우리 대응: 웹(Next.js App Router) 반응형을 기본 전제로, 데모 화면 폭 대응 확인. (Phase 6/7)
- **자동화 과신 → '내 감정이 빠진' 기록의 공허함**. → 우리 대응: 자동 채움은 메타데이터(시간·위치·날씨)에 한정하고, 감정·이야기는 AI 질문으로 사용자에게서 직접 끌어내 차별화. (Phase 5)

## 5. DB·핵심기능 설계에 즉시 반영할 결정

1. **`records` 자동 채움 필드 확정**: `captured_at`(EXIF 촬영시각), `lat`/`lng`(EXIF 좌표), `weather`(좌표·시각 기반 자동, nullable), `photo_url`(Storage). EXIF 없을 때 `manual_location` fallback 컬럼 포함. (Phase 2/3)
2. **후기 테이블 분리 유지**: `diaries`(비공개, `user_id` 본인만 RLS) ↔ `reviews`(공개 읽기·본인 쓰기, `place_id` FK)를 물리적으로 분리. 디로그식 '비공개 기본'을 RLS로 강제. (Phase 2)
3. **`places` 캐시 + 한글 검색 경로 보장**: TourAPI 확정 장소를 `places`에 캐시하고 `records.place_id`로 연결. 좌표 자동제안 실패 시 한글 키워드 검색으로 확정하는 경로를 필수 fallback으로 설계. (Phase 4)
4. **AI 질문 '건너뛰기' 기본 제공**: `records`에 메모만 있어도 일기 생성이 가능하도록, AI 질문은 선택적(skip 가능) 단계로. 입력 최소화(디로그 장점)와 감정 심화(우리 차별점)를 양립. (Phase 5/6)
5. **타임라인 정렬 키 확정**: 일기 생성 시 `records`를 `captured_at` → 장소 순으로 정렬해 묶는 규칙 고정(다수 사진 일괄 업로드 대응). (Phase 6)
6. **미디어 타입 확장 컬럼 예약**: `records.media_type`('photo' 기본)을 미리 둬 영상 확장 시 스키마 변경 최소화. (Phase 2)

## 6. 미검증·추가 확인 필요

- '사진 한 장 10초/3초 완성', '단계 수' 등 구체 수치 — 1차 출처 원문 미확인(검색 요약 기반). (low)
- 온보딩/첫 사용 흐름의 구체 단계 — JS 렌더링으로 본문 페치 실패, 추정. (low)
- 포토북·SNS용 이미지 카드 등 출력물 내보내기 기능 — 확인된 것은 URL 공유·파일 백업뿐, 존재/부재 단정 불가. (low)
- 원본/압축 화질 정책, 사진 필터·보정 기능 — 출처에서 확인 안 됨. (low)
- '혼자 여행' 단독 타깃팅 여부 — 디로그가 솔로트래블을 명시한 근거 없음(프라이빗 정서로 포괄). (low)
- 사용 지도 SDK(구글/애플/자체), 핀·라인 시각 스타일, 컬러·서체 등 비주얼 톤 — 텍스트 출처만으론 미확인(스크린샷 필요). (med)
- 정확한 누적 다운로드/MAU, 구독가 정확 구간, 부정 리뷰의 전수·비중 — AppBrain·Google Play 직접 열람 실패로 추정·표본 한계. (med)
