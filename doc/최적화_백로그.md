## 1. 이미지 저장 — 원본 유지 vs 압축  〔상태: 보류, 현행=원본/변환본 저장〕

### 현재 동작
- HEIC → 서버에서 JPEG로 변환(`/api/heic2jpeg`), 그 외는 원본 그대로 Storage 업로드.
- **추가 압축/리사이즈 없음.**
- **시각·장소는 `photos.captured_at/lat/lng` DB 컬럼에 저장** → 압축 여부와 무관하게 안전(EXIF는 변환 전 원본에서 먼저 추출).

### 핵심 사실
- iPhone 12MP 사진: HEIC ~2.7MB. **HEIC→JPEG 변환이 오히려 용량을 키움**(측정: 2.7MB → 4MB. HEIC가 더 효율적).
- Supabase 무료: **Storage 1GB / 대역폭 5GB·월** → 원본 유지 시 약 **200~300장**에서 한도 도달.

### 비교
| 기준 | 원본 유지(현행) | 압축(리사이즈+품질) |
|---|---|---|
| 화질 | 최상 | 표시·일기엔 충분, 약간↓ |
| 용량 | 장당 3~5MB(1GB 금방 참) | 장당 수백 KB(5~10배↑ 저장) |
| 대역폭 | 빨리 소진 | 절약 |
| 속도 | 업로드·표시 느림 | 빠름 |
| 시각·장소 | DB 저장(영향 없음) | DB 저장(영향 없음) |

### 추천(재검토 시작점)
- **압축 도입**: 업로드 전 클라이언트 리사이즈 **long edge ~2048px + JPEG 품질 0.8** → 장당 수백 KB, 표시 충분. **원본 미보관**.
- 구현: EXIF 추출(이미 앞단) → HEIC면 서버 변환 → 클라이언트 canvas 리사이즈/품질(또는 `browser-image-compression`) → 업로드.
- 더 욕심: **표시용 압축본 + 썸네일**(지도 마커/그리드용) 2단계. MVP엔 압축본 1개로 충분.

### 검토 트리거
- Storage 용량/대역폭 추이 모니터링, 데모 전 점검 〔CLAUDE §10〕.
- 결정 시 이 항목 갱신 + `CLAUDE.md` 정책 반영.
