# 혼자 여행자를 위한 AI 여행 일기 앱 — 기획 정리

> 한국관광공사 TourAPI 활용 공모전 출품작
> 인터뷰를 통해 확정된 내용 종합 (v4 — 기술 스택 확정·레퍼런스 분석 반영)

---

## 1. 한 줄 정의

**혼자 여행하는 사람**이 장소마다 **사진 + 한 줄 메모 + 짧은 질문 답변**을 남기면, AI가 그때의 상황과 감정을 정리해주고, 이를 바탕으로 **일기 작성을 보조**하는 앱.

핵심 강점은 **"AI 자동 정리 + 일기 작성 보조"**, 부가 가치는 **장소별 후기 게시판을 통한 관광 선순환**.

> **왜 혼행족인가:** 동행이 없어 그 순간의 감정을 나눌 상대가 없는 혼자 여행자에게 기록은 곧 동반자가 된다. AI가 질문을 던지는 방식이 혼자일 때 가장 자연스럽고, "혼자여서 더 잘 기록된다"는 차별점이 선명하다. 1인 여행 증가 트렌드와도 맞물린다.

---

## 2. 타겟 사용자

- **1차 주 타겟: 혼자 여행하는 사람 (혼행족)** — 국내 여행
- **추후 확장: 방한 외국인 관광객** — 다국어 기반 활용 (기획에 다국어 미리 반영)
- 공통: 글쓰기는 막막하지만 여행 기록은 남기고 싶은 사람

---

## 3. 핵심 사용자 흐름

```
[장소 도착]
   ↓
사진 업로드 (EXIF로 위치·시각 자동 인식 → 좌표를 먼저 추출·저장)
   ↓
TourAPI로 "이 장소 OO 맞나요?" 자동 제안 → 확정·수정
   ↓
한 줄 메모 작성 + AI가 카테고리별 질문 제시
 (혼행 정서 반영: "혼자 보는 이 풍경은 어땠나요?" / 인상적인 건? / 평점은?)
   ↓
AI가 답변을 그때의 상황·감정으로 정리 (코스 단위 = 장소 단위)
   ↓
정리된 재료들을 시간·장소순 타임라인으로 묶고 카테고리 태깅
   ↓
사용자가 이를 바탕으로 일기 작성 (AI 보조 + 문체 선택)
   ↓
[비공개] 일기는 늘 비공개 보관
[공개] 장소별 후기만 게시판에 공개 (혼자 다녀온 사람들의 같은 장소 후기로 묶음)
   ↓
[리마인더] "어제 다녀온 OO, 지금 정리해볼까요?" 푸시 + 이메일
```

## 4. 확정된 핵심 결정사항

| 항목 | 결정 |
|---|---|
| 주 타겟 | **혼자 여행하는 사람 (혼행족)**, 외국인은 추후 확장 |
| 코스 단위 | **한 장소 = 한 코스** (장소 단위) |
| 최소 입력 | 사진 + 한 줄 메모 + 질문 답변(감정·인상·평점 등) |
| 질문 방식 | TourAPI 카테고리별 **동적 질문** + **혼행 정서 반영** |
| 장소 인식 | **사진 EXIF 자동 인식** → TourAPI로 장소 제안 후 수정 가능 |
| 다국어 | UI + 일기 변환 + API 관광정보까지 **전부** (외국인 확장 대비) |
| 일기 문체 | **선택 기능 포함** — 담백 · 감성 · 유머 |
| 공개 범위 | **일기는 늘 비공개**, **장소별 후기만 공개** (혼행 후기로 묶음) |
| 외부 SNS 변환 | **후순위** (1차 범위 제외) |
| 사진 저장 | **원본 저장으로 시작** → 용량 추이 보며 추후 조정 |

## 5. TourAPI 활용 포인트 (공모전 핵심 평가 대상)

단순 자동완성에 그치지 않도록, API를 다음 4개 지점에 녹임:

1. **EXIF 좌표 → 장소 자동 인식** (위치기반관광정보 조회)
2. **장소 카테고리 데이터 → AI 질문 동적 생성** (자연/문화/맛집 구분)
3. **장소 공식 설명·정보 → AI의 "감정 환기" 재료** ("일출 보셨나요?")
4. **장소별 후기 게시판 → 사용자 후기 + 공식 관광정보 결합** (관광 선순환)

> 이 4개가 "TourAPI 없으면 안 되는 앱"이라는 근거가 됨.
> **검증 완료:** TourAPI 국문 관광정보 서비스는 위치기반관광정보·지역기반관광정보·키워드검색·소개정보·이미지정보 등 15종 약 26만 건을 제공. "EXIF 좌표 → 주변 장소 조회", "카테고리 분류", "공식 설명·이미지 제공"이 실제 API로 가능함을 확인.
> **공모전 규정:** TourAPI 활용은 필수. 1차 심사 전까지 웹 또는 앱으로 **실제 런칭(상용)** 되어 있어야 함 → 완성도·런칭이 곧 자격요건.

## 6. 기술 스택 (확정)

| 영역 | 선택 | 역할 / 비고 |
|---|---|---|
| 프론트 + 백엔드 | **Next.js** | 화면(프론트)과 서버 처리(API Route)를 한 프로젝트에서. 별도 백엔드 불필요 |
| 앱 출시 | **Capacitor 래핑** | 같은 Next.js 코드를 감싸 앱스토어·플레이스토어 앱으로. 웹·앱 단일 코드 |
| 서버 호스팅 | **Vercel** | Next.js를 인터넷에 배포. 서버리스 방식. 개인 프로젝트 무료 |
| DB·인증·스토리지 | **Supabase** | Postgres(DB) + Auth(로그인) + Storage(사진) 올인원. 무료로 시작 |
| 로그인 | **이메일+비밀번호 우선** | 소셜 로그인(구글·카카오)은 추후 |
| AI 모델 | **Gemini Flash** | 무료 티어. 멀티모달·함수호출·JSON·100만 토큰 지원 |
| 푸시 알림 | **웹 푸시 + 이메일 백업 → Capacitor 네이티브 푸시** | 아래 푸시 전략 참조 |

### 왜 이 조합인가 (한 줄 요약)
- **Next.js:** 코드 하나로 웹·앱 동시 제작, 화면+서버 한 몸, 팀이 이미 같은 구조(capacitor.config 확인) 사용.
- **Vercel:** 서버를 몰라도 자동 배포·관리, Next.js 제작사라 궁합 완벽, 무료.
- **Supabase:** 데이터·로그인·사진 저장을 한 곳에서, 직접 관리 불필요, 무료 시작.

### 앱 출시 경로 (웹 우선 → 추후 앱)
- **1차(공모전·초기): 웹 + PWA.** URL 하나로 심사·데모·설치 커버. 코드 단일.
- **추후: Capacitor 래핑.** Vercel에 배포한 Next.js 웹을 그대로 띄우는 구조로 스토어 출시. 카메라·EXIF·푸시는 네이티브 플러그인 처리.
- React Native는 웹을 따로 만들어야 하고 팀 자산을 못 써 1차 제외. (백엔드는 Supabase로 대체 가능하나 효율상 Next.js가 우위)

### 푸시 전략 (혼행 리마인더 = 리텐션 핵심)
- "AI가 상기시킴"이 곧 푸시 ("어제 다녀온 OO, 정리해볼까요?").
- iOS PWA 푸시는 iOS 16.4+부터 가능(홈 화면 설치 + standalone 필요, 도달률은 안드로이드보다 낮음).
- 우리 알림은 긴급성 낮은 정보성 리마인더라 웹 푸시로 충분.
- **약점 보완:** 핵심 리마인더는 푸시 + 이메일 두 채널.
- 앱 출시 후 Capacitor 네이티브 푸시(FCM/APNs)로 도달률 강화 가능 — RN 없이도 가능.

### 무료 티어 한도 (2026년 6월 확인 / 지속성 근거)
| 서비스 | 무료 한도 핵심 | 주의점 |
|---|---|---|
| Vercel (Hobby) | 월 대역폭 100GB, 함수 실행 100만 회, Active CPU 4시간, Blob 1GB | 한도 초과 시 과금 대신 **정지**. **비상업적 전용**(수익화 시 Pro $20/월). 함수 타임아웃 주의(긴 AI 응답) |
| Supabase (Free) | DB 500MB, 파일 1GB, 대역폭 5GB, 월 활성 5만 명, 프로젝트 2개 | **7일 무활동 시 자동 일시정지**(데모 전 깨워둘 것). 신규 프로젝트는 REST 접근에 Postgres grant 필요 |
| Gemini Flash | 분당·일일 요청 한도(수시 변동) | **Pro는 유료 전용**. billing 켜면 무료 사라짐. 무료 입력은 모델 개선에 사용될 수 있음 |

> 사진(스토리지 1GB·대역폭 5GB)이 가장 먼저 찬다. 데모는 원본 저장으로 충분하나, 실사용 확장 시 **업로드 전 클라이언트 이미지 압축(품질 85~90%)**이 거의 필수.

## 7. AI 작업 분류

| 작업 | 설명 |
|---|---|
| 질문 생성 | 사진·카테고리 기반 + 혼행 정서를 담은 질문 만들기 |
| 상황·감정 정리 | 메모 + 답변을 정리된 기록으로 변환 |
| 일기 변환 | 정리된 재료 → 일기 본문 (담백·감성·유머 문체 선택) |
| 다국어 처리 | 프롬프트로 처리, 별도 번역 API 불필요(1차) |

> 외부 SNS 변환은 후순위로 1차 범위에서 제외.
> AI 호출은 반드시 Next.js API Route(서버)에서 → Gemini 키 클라이언트 비노출.

## 8. 데이터 구조 방향 (초안)

공개·비공개 분리 결정에 따라 일기와 후기를 별도 테이블로 분리:

- **users** — 회원 (Supabase Auth 연동)
- **places** — TourAPI 장소 캐시 (장소명·좌표·카테고리·공식 설명)
- **records** — 코스 단위 기록 (사진 URL, 한 줄 메모, 질문 답변, AI 정리 결과) · **비공개**
- **diaries** — 일기 (records 기반, 문체 선택, AI 보조 본문) · **늘 비공개**
- **reviews** — 장소별 공개 후기 (좋아요, 간단 후기) · **공개**

### 사진 저장 방식 (DB와 Storage 분리)
- **실제 사진 파일** → Supabase **Storage**에 저장 (무거운 파일은 DB에 넣지 않음)
- **사진 위치(URL)·EXIF 좌표·시각** → **DB(records)**에 글자로 저장
- **순서 주의:** EXIF는 압축·리사이즈에서 사라질 수 있으므로 **① 좌표 먼저 추출·저장 → ② 사진 업로드 → ③ DB에 URL 기록 → ④ 좌표로 TourAPI 조회** 순서로 처리

## 9. 유사 서비스 레퍼런스 분석 (요약)

### 직접 경쟁 (콘셉트 거의 동일 — 반드시 의식)
- **디로그(D·LOG):** 사진 올리면 시간·위치·날씨 자동 채움 + 감정 기록 + 여행지도 + 웹·앱 동시. 우리와 가장 가까움. (단, AI 질문·일기 작성 보조·TourAPI·후기 게시판 없음)
- **스냅저니:** "사진 한 장 + 간단한 메모"로 감성 여행 일기. 입력 방식이 우리와 사실상 동일. (단, AI 정리·일기 보조·관광데이터 없음)

### 인접 (일부 기능 겹침)
- **어디(EODI):** GPS 경로 자동 기록 + AI 여행 플래너 + 다음 여행 추천. (경로·일정 중심, 감정·일기 아님)
- **Polarsteps:** 경로 추적 + 여행기 + 포토북 + 커뮤니티 팁. (글로벌·경로 기반, 국내 관광데이터·혼행 없음)
- **Journey.Cloud / Feeltracker:** AI 일기 도우미·감정 분석·멀티 플랫폼. (범용 일기, 여행 특화 아님)
- **Visited / 오킷 / PutPut:** 방문 장소 지도 수집·표시. (수집 목적, AI 정리·일기 아님)

### 수상작급 공통 패턴
입력 부담 최소화 · 자동 정리/시각화 · 멀티 플랫폼 · 지속 사용 유도 장치 · 변환/공유 출구 · 감정/회고 강조.

### 우리만의 빈틈 (아무도 동시에 하지 않는 3가지)
1. **AI가 질문으로 감정을 끌어낸다** — 경쟁 앱은 감정을 터치·아이콘으로만 받음
2. **TourAPI 관광 공식데이터를 일기 재료로 결합** — 일반 일기 앱엔 없음(공모전 평가 핵심)
3. **혼자 여행 특화 + 후기 선순환** — 어떤 앱도 혼행을 정면으로 내세우지 않음

> **포지셔닝:** "또 하나의 기록 앱"이 아니라 **"AI 질문 + TourAPI + 혼행 후기 선순환을 묶은 최초의 서비스"**. 단, 입력 최소화(질문 건너뛰기·기본값)는 경쟁 앱에서 반드시 벤치마킹.

## 10. 공모전 심사 기준 대비 점검

**1차 심사 (서면·기능심사, 100점)**
- 기획력(30): 혼행 타겟 + 3가지 결합 차별점 → 강화됨
- 완성도(30): 실제 작동·안정성이 관건 → **최대 리스크, 구현으로만 확보** (규정상 런칭 필수)
- 데이터 활용(20): TourAPI 4개 지점 활용 + 실제 가능 확인 → 강함
- 발전성(20): 혼행→외국인 확장·후기 선순환 → 보통, 지속성(비용 한도) 답변 보강 완료

**최종 심사 (PT, 100점)**
- 적정성(30): 문제·해결 논리 명확 → 강함
- 완성도(30): 라이브 데모 안정성 관건 → 리스크 (Supabase 일시정지 주의)
- 실용성(25): 입력 부담 최소화 근거 필요 (EXIF 자동·질문 건너뛰기)
- 발표(15): 별도 준비 영역

> **핵심 결론:** 양 심사 합산 60점이 "서비스 완성도"에 걸려 있음 → 끊김 없는 안정적 데모가 수상의 핵심.

## 11. 남은 확인 사항

- [ ] 입력 부담 최소화 설계 (질문 건너뛰기·기본값) — 레퍼런스 교훈 반영
- [ ] TourAPI 인증키 신청 (공공데이터포털 data.go.kr) 및 엔드포인트별 응답 필드 확인
- [ ] 데모 전 Supabase 프로젝트 깨워두기(7일 일시정지 방지) — 라이브 시연 대비

## 13. 확정 기술 스택 (변경 금지)

- **프론트+백엔드:** Next.js (App Router). 별도 백엔드 프레임워크(Nest/Express 등) 사용하지 않음. 서버 로직은 Next.js API Route(또는 Route Handler)에서 처리.
- **앱 출시:** Capacitor로 웹 코드를 래핑(웹·앱 단일 코드). React Native 사용하지 않음.
- **호스팅:** Vercel (서버리스).
- **DB·인증·스토리지:** Supabase (Postgres + Auth + Storage).
- **AI:** Google Gemini Flash (무료 티어, billing 비활성 프로젝트 유지).
- **로그인:** 이메일+비밀번호 우선. 소셜 로그인은 추후.

> Claude Code는 위 스택을 임의로 바꾸거나 대체 라이브러리를 제안하지 말 것. 다른 스택이 더 낫다고 판단되면 변경하지 말고 주석/메모로만 의견을 남길 것.

## 14. TourAPI 호출 규칙 (가장 자주 틀리는 부분)

- **반드시 신버전 `KorService2` 사용.** 구버전 `KorService1` 및 오퍼레이션 접미사 `1`(locationBasedList1 등)은 사용 금지.
- 사용할 오퍼레이션: `locationBasedList2`(좌표 주변 조회), `detailCommon2`(공통/개요), `detailIntro2`(운영시간·요금), `detailImage2`(이미지), `areaCode2`(지역코드 조회), `searchKeyword2`(키워드 검색).
- **호출은 반드시 서버(API Route)에서.** `serviceKey`를 클라이언트 번들에 절대 노출하지 말 것.
- **좌표 파라미터 순서 주의:** `mapX = 경도(longitude)`, `mapY = 위도(latitude)`. (구글맵 표기는 위도,경도 순이라 반대 — 바꿔 넣으면 엉뚱한 위치 조회)
- 위치 기반 조회는 좌표만 있으면 되므로 지역코드 불필요. 지역코드는 "특정 지역 전체 목록" 같은 별도 기능에서만 areaCode2로 받아 사용.
- `_type=json`, `MobileOS`, `MobileApp`은 매 요청 필수.
- 위치 조회 기본 호출 예시(서버 측):
  ```
  https://apis.data.go.kr/B551011/KorService2/locationBasedList2
    ?serviceKey={DECODED_KEY}&MobileOS=ETC&MobileApp=TripDiary&_type=json
    &mapX={경도}&mapY={위도}&radius=2000&numOfRows=20&pageNo=1&arrange=E
  ```
- contentTypeId(타입 필터): 12=관광지, 14=문화시설, 15=축제공연행사, 28=레포츠, 32=숙박, 38=쇼핑, 39=음식점.
- TourAPI 응답은 데이터가 없으면 해당 key 자체가 빠질 수 있음 → 모든 필드 접근은 optional/방어적으로 파싱할 것.

## 15. 보안·환경변수 규칙 (사고 방지)

- 모든 키는 `.env.local`에 저장하고 git에 커밋 금지(`.gitignore` 확인).
- **클라이언트 노출 가능:** Supabase URL, Supabase anon key (NEXT_PUBLIC_ 접두사 허용).
- **절대 클라이언트 노출 금지:** Supabase `service_role` key, Gemini API key, TourAPI `serviceKey`. → 이 키가 필요한 작업은 전부 서버(API Route/서버 컴포넌트)에서만.
- **Supabase RLS(Row Level Security) 필수.** 테이블 생성 시 RLS를 켜고 정책을 명시:
  - `records`, `diaries`: 본인(`auth.uid()`)만 읽기/쓰기. **타인 접근 불가.**
  - `reviews`: 누구나 읽기 가능(공개), 작성·수정은 본인만.
  - `places`: 읽기 공개(캐시), 쓰기는 서버만.
  - RLS 누락 시 남의 일기가 노출되는 심각한 사고 → 테이블마다 정책 반드시 확인.

## 16. 핵심 로직 순서 & MVP 범위

### 사진 업로드 처리 순서 (반드시 이 순서)
1. 클라이언트에서 사진의 **EXIF 좌표·촬영시각 먼저 추출** (압축/리사이즈 전에).
2. 추출한 좌표·시각을 서버로 전송해 DB(records)에 저장.
3. 사진 파일을 **Supabase Storage**에 업로드.
4. Storage가 반환한 **URL을 DB(records)에 기록** (사진 원본은 DB에 넣지 않음).
5. 좌표로 서버에서 **TourAPI locationBasedList2 호출** → 주변 장소 후보 반환 → "이 장소 OO 맞나요?" 제안.
> EXIF는 이미지 가공 시 사라질 수 있으므로 1번이 반드시 최우선.

### AI 호출 (Gemini Flash, 서버에서만)
- 작업: ① 카테고리별 질문 생성 ② 메모+답변 → 감정·상황 정리 ③ 일기 본문 생성(문체: 담백/감성/유머 선택) ④ 다국어 출력.
- 응답이 길어질 수 있으니 일기 생성은 **스트리밍** 고려(Vercel 함수 타임아웃 대비).

### 1차(공모전) MVP 범위 — 욕심내지 말 것
- **포함:** 회원가입/로그인(이메일+비번), 사진 업로드+EXIF, TourAPI 장소 제안, AI 질문→정리→일기 작성(문체 선택), 일기 비공개 보관, 장소별 공개 후기(좋아요·간단 후기), 리마인더(웹 푸시 또는 이메일).
- **제외(후순위):** 외부 SNS 변환, 소셜 로그인, 외국인 다국어(구조만 대비), 네이티브 푸시(앱 출시 후).
- 범위를 임의로 키우지 말 것. 완성도(끊김 없는 핵심 흐름)가 수상의 핵심.

### 운영 주의
- Supabase 무료는 7일 무활동 시 일시정지 → **데모 전 반드시 깨워둘 것**.
- 사진은 원본 저장으로 시작하되 용량 모니터링 → 초과 조짐 시 업로드 전 클라이언트 압축(품질 85~90%).
- **1차 심사 전까지 실제 런칭(상용) 필수** = 자격요건.
